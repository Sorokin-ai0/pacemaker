# DECISIONS.md — assumptions made instead of asking

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Build in `/Users/iaroslavsorokin/half marathon` (empty dir created today), not the `twitter product` dir. | Prompt says "this repository"; the half-marathon-named workspace dir is clearly the intended home, while `twitter product` holds unrelated demo apps. Will `git init` here. |
| 2 | TypeScript on both server and client. | Spec doesn't name a language; shadcn/ui is TS-first and "production-quality" + zod pair naturally with TS. ESM throughout as required. |
| 3 | `longRunDay` stored as Int 0–6, 0 = Sunday (JS `Date.getDay()` convention). | Avoids locale ambiguity; UI shows weekday names. |
| 4 | Regenerating a plan deletes the old plan and its workouts; `LoggedRun.plannedWorkoutId` is set null (`onDelete: SetNull`). | Keeps "one active plan" simple in v1; run history is never lost. |
| 5 | 14-mi overreach week applies to Intermediate/Advanced only; Beginners hold their 12-mi cap. | A 14-mi run two weeks out is an injury risk for someone capped at 12 mi; spec sentence is ambiguous, safety wins. |
| 6 | Weekly template fixed at 1 long + 1 quality + 3 easy + 2 rest (= 7 days). | Only combination satisfying "2–3 easy, 1–2 rest" with every day assigned. Quality alternates tempo/speed week to week. |
| 7 | "Ambiguous plan length → 14 weeks" triggered when race date is missing, unparseable, or in the past; race date then set to today + 14 weeks. | Spec defines the default but not the trigger conditions. |
| 8 | Riegel basis run = fastest-pace logged run with distance ≥ 8 km in the last 60 days. | "Best recent long run" needs an operational definition; ≥ 8 km filters out short runs that inflate projections. |
| 9 | Dark-mode preference stored client-side (localStorage), unit preference server-side on User. | Unit affects API-adjacent display everywhere and is part of the given data model; theme is device-level cosmetic. |
| 10 | devops-agent authors Dockerfile/railway.json during wave 3 (files only); actual deployment steps happen after the pause-2 go-ahead. | Prompt both schedules devops-agent early and places "deployment" after pause 2 — resolved as: artifacts early, deploy actions late. |
| 11 | `bcrypt` (native) rather than `bcryptjs`; Docker base image `node:20-slim` (glibc) so prebuilt binaries work. | Spec names bcrypt explicitly; slim avoids alpine/musl native-module friction. |
| 12 | Demo credentials: `demo@pacemaker.run` / `Demo1234!`. | Needed something; documented in README. |
| 13 | Accent color: electric blue (sky-400/500 family). | Spec offered deep teal or electric blue; blue reads better on the dark card UI and is unmistakably not Strava orange. |
| 14 | Pinned Prisma ^6 and Express ^4 even though Prisma 7 / Express 5 are the latest majors. | Both new majors change config/middleware semantics; v6/v4 are the battle-tested surfaces this plan assumes. Prisma's `package.json#prisma` seed hook deprecation noted for a future Prisma 7 upgrade. |
| 15 | Plan-length clamping anchors the plan END at race date: `startDate = raceDate − totalWeeks` weeks. If the race is < 8 weeks out, early plan weeks fall in the past (shown as missed); if > 20 weeks out, the plan starts in the future. | Clamping to 8–20 weeks can't both start today and end at the race; anchoring to the race keeps taper/race-week placement physiologically correct. |
| 16 | `POST /api/auth/logout` requires no auth and always returns 204 (clears cookie). | Idempotent logout is friendlier to a frontend holding an expired session; contract's error column for logout was empty. |
| 17 | `GET /api/stats` with no plan returns 200 with `weeklyMileage: []`, zeroed adherence, `countdown: null`, `taper: null`; paceTrend/projection still computed from logged runs. | Runs exist independently of plans; dashboard can render partial data instead of erroring. |
| 18 | Weekly volume growth capped at min(2× starting volume, 50/70/90 km for beg/int/adv). | Spec required a "sane" cap without a number; these are conventional half-marathon ceilings. |
| 19 | Cutback week = 80% of the previous week; the following week resumes the pre-cutback trajectory rather than re-ramping from the cutback value. | Standard periodization: recovery weeks shouldn't permanently lower the ramp. |
| 20 | API semantics: emails normalized (trim+lowercase); date inputs accept full ISO timestamps by truncating to the date; runs `to` filter inclusive; `daysToRace` may go negative after the race; weekly `plannedKm` groups by plan `weekIndex` (rescheduled workouts stay in their week) while `loggedKm` groups by calendar date; `race`-type workouts count toward adherence; seed forces its most recent long run to ≥ 14 km so the Riegel projection has a realistic basis mid-plan. | Operational details the contract left open. |
| 21 | UI treats `weekIndex` as 0-based and displays "Week N+1 of totalWeeks"; the "current week" is derived from the workout nearest today. | Matches generator output; robust when the plan starts in the past/future (see #15). |
| 22 | Switching a workout's type to `rest` in the calendar editor sends `targetDistanceKm: null`. | Rest workouts carry null distance; backend must accept explicit null on PATCH (verified during integration). |
| 23 | Run form offers "nearby" planned workouts (non-rest, within ±3 days, not already fulfilled) as fulfillment options, defaulting to same-day; countdown renders "Race day" instead of negative days; a profiled user with no plan gets an empty state linking to onboarding rather than a redirect loop. | UX details the spec left open. |
| 24 | **Pivot to local-storage preview build** (user-directed): client persistence moved to a localStorage adapter (`client/src/local/`) implementing the exact API_CONTRACT.md surface; local-only passwordless "auth"; blank first-run state; demo seed disabled by default (opt-in via `SEED_DEMO=true`); `npm run dev` = client only (`dev:full` = API+client). Server, Prisma schema, and 66-test suite kept intact for the swap-back. | User has no database set up yet and wants the full UI clickable; adapter keeps the real backend a drop-in replacement. |
| 25 | Preview additions: plan generator ported client-side with a `restDaysPerWeek` (1–2) option; AI Coach scaffold (`getCoachAdvice()` mock + dashboard CoachCard) with a TODO seam for a real LLM call; display settings (pace↔speed, HR fields on/off) in a SettingsProvider; "Reset all local data" in Settings. | Requested feature set for flow testing; interfaces kept stable so real implementations drop in. |
| 26 | Fixed run-logging 400: Radix Select can emit `""` when its selected option unmounts; client now normalizes `""/none → null` on submit and the server zod schema preprocesses `"" → null` for `plannedWorkoutId`. | Root cause of the Wave-4 integration bug; fixed on both sides defensively. |
